# 対話形式ストーリーの実例パターン集

このファイルは SKILL.md の補助資料。具体的な型を持ちながら執筆するときに参照する。

---

## 1. プロローグのパターン

### パターン1: エラーから始まる（GCP IAM 編）
```
**あらき**: 「先輩、助けてくださいぃ……」
**桐生**: 「お、あらきさん。今日はどうした？」
**あらき**: 「Cloud Run にサービスをデプロイしようとしたら、こんなエラーが出て……」
→ 具体的なエラーメッセージを提示 → なぜこうなるかを全章で解き明かす
```

### パターン2: ドキュメントが読めないから始まる（DNS 編）
```
**あらき**: 「Route53 が悪さをしてて、ゾーンがどうのこうの……ぜんぶカタカナです……！」
→ 「ぜんぶカタカナ」状態から「自分の言葉で説明できる」状態まで登る
```

### パターン3: コマンドの意味が分からないから始まる（CIDR 編）
```
**あらき**: 「ハンズオンで `--cidr-block 10.99.0.0/16` って打ちましたよね。
あれ、意味が分かってなくてコピペしただけでした……」
→ コピペしていた値の意味を全章で解き明かす
```

### パターン4: 業務上のやり取りから始まる（IAM 実践編）
```
**あらき**: 「パートナーに GCP アクセスを提供したいんですけど、
サービスアカウントは作れたのにロールを振れなくて……」
→ 実際のチャットのやり取りを分析して解決策を導く
```

### プロローグに含めるべき要素（必須）

- **具体的な困りごと**（エラー画面・コマンド・チャット）
- **このストーリーで到達するゴール宣言**（桐生先輩が口頭で）
  - 例: 「終わったら、新人くんに DNS の階層構造を5分で説明できるようになる」
- **これから何章で何を学ぶかの予告**（簡潔に）

## 2. 概念マップ提示のパターン

プロローグ直後または第1章末で必ず提示する。

```
**桐生**: 「いきなり細部に入る前に、これから登る山の全体像を見ておこう」

[ここで Mermaid 図またはテキスト図]

**桐生**: 「今日はまず A の山に登る。次に B。最後に A と B が
            実は C という同じ思想で繋がっていることを見るよ」
```

各周回の節目で再提示して「今ここ」を示す。

## 3. あらきの例え話パターン

良い例え話の条件: **日常で誰もが体験すること** を使い、**技術概念の核心** を突く

| 技術概念 | あらきの例え | ポイント |
|---|---|---|
| DNS | 電話帳 | 名前→番号の変換 |
| SNI | コンサートのもぎり | チケット（SNI）を見て案内先を決める |
| DNS 偽装 | カフェで中身のコーヒー豆をこっそり変える | メニュー名（FQDN）は同じだが中身（IP）が違う |
| SA と actAs | 車と鍵 | 車（SA）の機能と、鍵（actAs）を渡す行為は別 |
| 特権昇格防止 | 階段に鍵をかける | 権限の階段を一段飛ばしさせない |
| SA 作成 vs ロール付与 | 弁当箱 vs おかず | 器は作れるが中身を入れる権限がない |
| キャッシュ | おばあちゃんの引き出し | よく使うものは手前にしまっておく |
| 非同期処理 | ラーメン屋の注文票 | 並ばず席に座って番号を待つ |
| トランザクション | 銀行の振り込み | 全部成功 or 全部なかったことに |

**重要**: 第1周回で出した例え話は、第2周回で **「実はその例えはここまでしか正しくない」と裏切る** ことで深さが出る。

## 4. 桐生先輩の格言パターン

- 「**意図通りに壊れている**」 — 仕様通りの挙動が問題に見えるとき
- 「**エラーは敵じゃなくて、設計者からのメッセージ**」 — PERMISSION_DENIED の理解
- 「**分からないと言えるのは才能だ**」 — 素直さの価値
- 「**設計には必ずトレードオフがある**」 — 銀の弾丸はない
- 「**ネットワークの知識は全部地続きだ**」 — DNS も CIDR もルーティングもひとつの地図
- 「**抽象化は理解の最短ルートだが、過信すると詰む**」 — レイヤーを剥がす重要性
- 「**動いているコードには理由がある**」 — レガシーの尊重
- 「**速さは正しさの後に来る**」 — 最適化の優先順位

## 5. 誤解の予防接種パターン

各章で必ず実施。以下のテンプレートを使う。

```
**桐生**: 「さて、ここで多くの人がこう誤解するんだけど、
            あらきさんはどう思う？」
**あらき**: 「えーっと、たぶん〇〇ですよね？」
**桐生**: 「(笑) 半分正解。でもそれだと △△ のときに破綻する。なぜなら〜」
```

**先に誤解を経験させて、後で訂正する** ことで、誤解の予防接種になる。

## 6. 「なぜ」の5階層を掘るパターン

```
表面: 「DNS は名前を IP に変換する」
  ↓ なぜ階層構造？
  → 「単一サーバでは全インターネットの名前を管理できないから」
    ↓ なぜ管理できない？
    → 「数十億のドメインがあり、更新も頻繁で、単一障害点になるから」
      ↓ なぜ階層という選択肢？
      → 「権限の委譲を可能にするため。各組織が自ドメインを管理できる」
        ↓ 他の選択肢はあった？
        → 「フラットなDHT（分散ハッシュテーブル）案もあった。Namecoin等」
          ↓ 将来は？
          → 「DNSSECやDoH/DoT、ブロックチェーン名前空間の動きもある」
```

各章で最低3階層、できれば5階層まで掘る。

## 7. 反例・トレードオフ提示パターン

```
**桐生**: 「ここまで聞くと完璧な仕組みに見えるけど、実は弱点がある」
**あらき**: 「えっ、どこですか？」
**桐生**: 「〇〇のケースでは破綻する。〜〜という設計と比べると、
            この仕組みは ✕✕ を犠牲にして △△ を取っている」
**あらき**: 「あ、銀の弾丸じゃないんですね……」
**桐生**: 「(笑) 設計には必ずトレードオフがある」
```

## 8. 章末ミニ演習のパターン

> **折りたたみ解答の書式（Obsidian 必須ルール）**: `<details>` は引用（`>`）の中に入れない。前に空行を置き、`<details>`/`<summary>…</summary>`/`</details>` を各行に分け、`<summary>` 直後と `</details>` 直前に空行を入れる。詳細と正誤例は SKILL.md「4-6-1. 折りたたみの書き方」を参照。

### 種類1: 自分の言葉で説明する
```markdown
### 章末ミニ演習

> 問1: この章で学んだ「○○」を、技術用語を一切使わずに小学生に説明してください。

<details>
<summary>解答例</summary>

[解答をここに記載]

</details>
```

### 種類2: 未知のシナリオに適用する
```
問2: あなたが新しい SaaS を立ち上げるとき、〇〇の仕組みをどう設計しますか？
理由とともに答えてください。
解答例: [複数の正解パターンを提示]
```

### 種類3: あえて間違った設計を批評する
```
問3: 以下の設計案は何が間違っていますか？
[設計案の提示]
解答例: [どこが破綻するかを示す]
```

### 種類4: 絵を描く
```
問4: ここまでに学んだ概念を、A4一枚の図にまとめてください。
ヒント: [主要要素を列挙]
```

### 種類5: 既習概念との関連付け
```
問5: 第3章で学んだ「△△」と、この章の「○○」はどう関連していますか？
```

## 9. 章間チェックポイントのパターン

2〜3章おきに配置する。

```
## 中間チェックポイント — ここまでの整理

**桐生**: 「ここで一度、ここまでの内容を整理しよう。
            あらきさん、第1〜3章で学んだことを、僕に説明してみて」
**あらき**: 「えーっと、まず○○があって、次に△△が……」
**桐生**: 「(うなずきながら聞く) ……うん、概ね合ってる。
            ただ一つだけ、〇〇のところは ✕✕ の理解が浅い。
            なぜなら〜」
**あらき**: 「あ、そうか。もう一度振り返ってみます」

[再提示: 概念マップで「今ここ」を示す]
```

## 10. 終章のパターン（強化版）

必ず以下の構造を持つ:

```
## 終章 — あらき、自分の言葉で語る

**桐生**: 「それじゃあ最後に、会議で他チームに話すつもりで全体を説明してみて」

[あらきが地の文で一気に説明（数百字）。箇条書きではなく流れのある説明]

**桐生**: (数秒の沈黙) 「……うん。合格だ」
**あらき**: 「やったー！」
**桐生**: 「ただ、一つだけ補足させて。〇〇のところは△△の方が正確だ」

### 教える練習

**桐生**: 「次に、明日入ってくる新人くんに同じ内容を教える練習をしよう。
            プログラミング初めての人にも分かるように、5分で」

[あらきが超初心者向けに同じテーマを再説明する]

### 上級者の盲点

**桐生**: 「最後にひとつだけ、ベテランでも引っかかる罠を見せておく」

[上級者でも引っかかる落とし穴を1つ提示]

### 次のテーマへ

**桐生**: 「これを学んだ次は ○○ に進むと、ここで張った伏線が
            別の角度で回収されるよ」
```

## 11. 応用シナリオ集（付録）のパターン

```
## 付録: 応用シナリオ集

### シナリオ1: [具体的な業務状況]
状況: [描写]
問: あなたなら何を確認し、どう判断しますか？
解答例: [複数の妥当解と理由]

### シナリオ2: [トラブルシューティング]
状況: [エラーログや現象]
問: 根本原因を推測してください。
解答例: [仮説の立て方と検証手順]

### シナリオ3: [設計判断]
状況: [複数の選択肢が提示される]
問: どれを選びますか？ 制約条件によって答えがどう変わるかも述べてください。
解答例: [トレードオフ分析]
```

最低3個、上級テーマでは5個以上。

## 12. 良いストーリーと悪いストーリーの違い

| 観点 | 悪い例 | 良い例 |
|---|---|---|
| 章数 | 5章で終わる | 10〜15章で深掘り |
| 1章の長さ | 300字でまとまる | 1000〜2000字 |
| 周回 | 単線的 | 直感→構造→設計の3周 |
| 質問 | 「で、これはどうですか？」 | 「言われてみると、なぜ〇〇は△△じゃないんですか？」 |
| 例え話 | 一度だけ出して終わり | 第1周で出し、第2周で裏切り、第3周で精緻化 |
| トレードオフ | 触れない | 各章で明示 |
| 誤解 | 出さない | 章ごとに先回りで提示 |
| アウトプット | 末尾Ankiのみ | 各章末ミニ演習 + 章間チェック + 教える練習 |
| 歴史背景 | 触れない | 要所で発明者の意図・業界文脈 |
| 終章 | あらきの説明で終わり | 説明 + 教える練習 + 盲点 + 次への橋渡し |

悪い例は **「短くまとめる癖」の症状**。良い例の方向に常に押し進める。
