# Spec Review

## 目的

コードの変更が依拠する **Web 標準仕様・言語仕様・ライブラリ仕様** を権威あるソースから調査し、実装が仕様に準拠しているか、業界のベストプラクティスに沿っているかを検証する。

`/code-review` との違い: `/code-review` はコード品質・セキュリティの静的チェック。このスキルは**外部ソースを能動的に調査**して仕様違反やベストプラクティス乖離を発見する。

## 入力

- 対象: 現在のブランチの差分（`git diff`）、または指定されたファイル群
- オプション: 特に調査すべき技術領域の指定（例: "SSE の仕様準拠性を重点的に"）

## 手順

### Phase 1: 技術領域の特定

対象コードを読み、実装が依拠する技術領域を抽出する。

抽出する観点:
- **Web API / ブラウザ API**: fetch, ReadableStream, EventSource, AbortSignal, WebSocket, IndexedDB 等
- **プロトコル / データ形式**: SSE, HTTP/2, JSON, MessagePack, Protocol Buffers 等
- **言語機能**: AsyncGenerator, Proxy, WeakRef, Iterator Helpers, Import Attributes 等
- **ライブラリ固有の API**: React hooks ルール、Express middleware 契約 等

### Phase 2: 権威あるソースの調査

特定した技術領域ごとに、以下の優先順位でソースを調査する。Agent ツールで並列に調査エージェントを起動する。

#### 調査ソースの優先順位

1. **公式仕様書** — WHATWG (HTML, Streams, Fetch, URL), W3C, ECMA-262, TC39 提案, RFC
2. **公式ドキュメント** — MDN Web Docs, Node.js Docs, React Docs, ライブラリ公式ドキュメント
3. **著名 OSS の実装** — 同じ技術を扱う広く利用されているライブラリのソースコード
4. **SDK / リファレンス実装** — Anthropic SDK, OpenAI SDK, Vercel SDK 等の大手提供の参照実装

#### 調査項目

各ソースから以下を収集する:

- 仕様が要求する必須の処理（MUST / SHOULD / MAY の区別）
- エッジケースの処理ルール（仕様書の注記、著名ライブラリの特殊処理）
- リソース管理パターン（メモリリーク防止、接続クリーンアップ）
- ブラウザ間の互換性差異
- セキュリティ上の考慮事項

### Phase 3: 照合レビュー

調査結果と実装コードを照合し、以下を検出する。Agent ツールでコードレビューエージェントを起動する。

| カテゴリ | 説明 |
|---------|------|
| **仕様違反** | 公式仕様の MUST 要件を満たしていない箇所 |
| **ベストプラクティス乖離** | 著名 OSS / SDK が共通して採用するパターンからの逸脱 |
| **エッジケース未処理** | 仕様書やライブラリが対応しているが実装が見落としているケース |
| **互換性リスク** | 特定ブラウザ / ランタイムで動作しない可能性 |
| **セキュリティ懸念** | 仕様やライブラリが警告する安全でないパターン |

### Phase 4: 報告

以下の形式で報告する:

```
## 調査したソース
- [ソース名](URL) — 調査理由

## 発見事項

### 重大度: 高
| # | ファイル | 問題 | 根拠ソース |
|---|---------|------|-----------|

### 重大度: 中
...

### 重大度: 低
...

## 準拠が確認された項目
（正しく実装されている点も明示する）
```

各発見事項には必ず**根拠となるソースの URL またはライブラリ名 + ファイルパス**を付記する。根拠なしの指摘は報告しない。

## 完了条件

- [ ] 対象コードが依拠する技術領域が全て特定されている
- [ ] 各技術領域について最低 1 つの権威あるソースを調査している
- [ ] 発見事項に全て根拠ソースが付記されている
- [ ] 重大度が高い問題について修正案が提示されている
