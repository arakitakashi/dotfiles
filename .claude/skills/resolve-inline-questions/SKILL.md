# Resolve Inline Questions

ドキュメント内の `//` プレフィックス疑問コメントを特定し、調査・回答して本文スタイルに合わせた回答で置換する。

## ワークフロー

### Phase 1: 疑問の特定

1. 対象ファイルを Read で読む（大きいファイルは分割読み取り）
2. `grep -n "^//" <file>` で全 `//` 行を一覧する
3. **コードブロック内のコメント**（``` で囲まれた中の `//`）を除外し、**本文中の疑問コメントだけ** を抽出する
   - 疑問コメントの判定基準: コードブロック外にあり、`?` `ですか` `ください` `知りたい` `教えて` 等の疑問表現を含む、または文脈上質問として書かれている
4. 疑問リストを行番号付きで一覧し、件数をユーザーに報告する

### Phase 2: 疑問の分類

各疑問を以下のカテゴリに分類する:

| カテゴリ | 基準 | 対応方法 |
|---|---|---|
| **調査系** | 「一次ソース」「OSS事例」「推奨されているか」「信頼できるソース」など外部情報の裏付けを求めている | Agent を並行起動してリサーチ |
| **概念系** | 「〜とは何か」「仕組みを教えて」「詳しく解説して」など概念の理解を求めている | 直接回答を構成する |
| **判断系** | 「〜すべきか」「〜のほうが良いのでは」など設計判断への意見を求めている | トレードオフ比較表で回答する |

### Phase 3: 調査の実行

- **調査系の疑問**: Agent ツールで並行リサーチを起動する（`run_in_background: true`）
  - 1エージェント1疑問、プロンプトに具体的な調査指示を含める
  - 回答は400語以内で、URL・日付・具体的事実を求める
- **概念系・判断系**: 調査エージェントの完了を待つ間に回答を準備する

### Phase 4: 回答の構成

**文体の検出と適応** — 対象ファイルの既存の文体に合わせる:

- **対話形式** (`**名前**: 「〜」` パターン): 登場人物の対話として回答を構成する。質問者キャラが `//` の疑問を口にし、回答者キャラが答える形式にする
- **技術文書** (見出し・箇条書き中心): `> **補足:**` ブロックか新しいセクションとして挿入する
- **ノート/メモ** (簡潔な記述): 簡潔な回答を `//` の直後に挿入する
- **判定不能**: そのまま Markdown の段落として挿入する

**回答の品質基準**:
- 一次ソースがある場合は必ず URL を添える
- 比較が含まれる場合はテーブルを使う
- コード例は既存のコードブロックのスタイル（言語タグ・コメント形式）に揃える
- 調査結果で「元の記述が不正確」と判明した場合は、補足として正確な情報を提示する（元の記述は尊重し、削除しない）

### Phase 5: 挿入と検証

1. Edit ツールで各 `//` コメント行を回答で置換する（`old_string` に `//` 行の全文を指定）
2. 独立した箇所は並行で Edit する
3. 全挿入後に `grep -n "^//" <file> | grep -v "^[0-9]*:// "` で疑問コメントの残存を確認する（コードブロック内の正当な `//` コメントは除外）
4. 対応結果の一覧表（箇所・疑問・対応内容）をユーザーに報告する

## 注意事項

- `//` の後にスペースがある行（`// コメント`）はコードブロック内のコードコメントである可能性が高い。コードブロック外かつ疑問表現を含むもののみ対象とする
- 大量の疑問（10件超）がある場合は TaskCreate で進捗管理する
- ファイルが対話形式の場合、登場人物名・口調・敬語レベルを既存の対話から正確に模倣する
