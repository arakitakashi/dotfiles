# Book Translator - 技術書翻訳スキル

技術書の英語原文を、原文に忠実かつ技術的に本質がわかりやすい日本語に翻訳する。

## 最重要原則: 原文忠実性の保証

**絶対に要約しない。** 原文のすべての段落、すべての文、すべての箇条書き項目を翻訳する。「原文を参照してください」のような省略は禁止。コードブロック、テーブル、図のキャプションもすべて含める。

翻訳品質の検証基準: **翻訳後の行数が原文行数の80%未満の場合、その翻訳は失敗とみなす。** 日本語は英語より行数が少なくなることがあるが、80%を大幅に下回ることは要約が発生している証拠。

## ワークフロー

### 1. 対象ファイルの分析

```bash
grep -n "^# 第" "<filepath>"        # 翻訳済み章
grep -n "^Chapter [0-9]" "<filepath>" # 未翻訳の英語章
wc -l "<filepath>"                    # 総行数
```

各章の行範囲と行数を記録する。これが後の品質検証の基準になる。

### 1.5. ソース品質検査（推奨・翻訳前）

Webから抽出されたmarkdown原文には、コードブロック破損や装飾ラベルの過剰な見出し化が起こりうる。翻訳前に検査する。

```bash
FILE="<原文ファイル>"

# コードブロック内の「数字だけの行」（行番号連結の典型）
COLLAPSED=$(awk '/^```/{c++; if(c%2==1){inblock=1; next} else {inblock=0; next}} inblock && /^[[:space:]]*[0-9]+[[:space:]]*$/' "$FILE" | wc -l)
echo "コードブロック内の数字のみの行: $COLLAPSED"

# H5 装飾ラベル（目次を破綻させる）
echo "##### 装飾ラベル: $(grep -cE '^##### (Listing|Figure|Table|NOTE|TIP|WARNING)' "$FILE")"
```

**問題が見つかった場合:**
- 抽出元が **Manning livebook / MEAP** → 専用スキル `manning-translator` を使う（コード救済スクリプトや装飾ラベル正規化が含まれる）
- それ以外のソース → ユーザーと方針を協議:
  1. 原文を再抽出
  2. コード欠損部分には `<!-- 注: 原文ではここに数行のコードが含まれていたが、抽出エラーのため省略 -->` を挿入
  3. 翻訳後に文脈から復元

### 2. 翻訳単位の決定（最重要ステップ）

**1つのAgentに渡す原文は最大200行とする。** これを超えるとコンテキスト圧縮で要約が発生するリスクが高い。

分割ルール:
- 200行以下の章 → 1 Agent で翻訳
- 201〜400行の章 → 2 Agent に分割（セクション境界で分割）
- 401行以上の章 → 200行以下の単位に分割（セクション境界で分割）
- **1つの章を複数Agentに分ける場合、必ずセクション見出し（`##` や空行）の境界で分割する**

例: 650行の章 → 3つのAgent（約200行 + 230行 + 220行）

### 3. ファイル分割

```bash
cd "<file_directory>"
head -n <翻訳済み最終行> "<filename>" > temp_head.md
tail -n +<翻訳後テキスト開始行> "<filename>" > temp_tail.md
```

### 4. 翻訳の実行

各Agentへの指示テンプレート（`references/translation-rules.md` の「Agent指示テンプレート」を参照）を使用する。

**Agentへの指示に必ず含める情報:**
1. 翻訳対象の正確な行範囲（開始行〜終了行）
2. 翻訳ルールの全文（テンプレートから）
3. **原文の行数と、翻訳後に期待される最低行数（原文行数の80%以上）**
4. **「要約するな、すべての段落を翻訳せよ」という明示的な警告**
5. 翻訳結果をファイルに書き込む指示（temp_body_N.md）

**並列実行:**
- 独立した章・セクションは並列Agentで同時翻訳
- 各Agentは `temp_body_1.md`, `temp_body_2.md` ... のように番号付きファイルに出力
- 全Agent完了後に番号順で結合

### 5. 品質検証（必須）

翻訳完了後、**結合前に**各翻訳チャンクの品質を検証する。

```bash
# 各翻訳チャンクの行数を検証
for f in temp_body_*.md; do
  echo "$f: $(wc -l < "$f") 行"
done
```

**検証基準:**
- 翻訳行数 ÷ 原文行数 ≧ 0.80 → OK
- 翻訳行数 ÷ 原文行数 < 0.80 → **再翻訳が必要**
- 翻訳行数 ÷ 原文行数 < 0.50 → **深刻な要約。そのチャンクをさらに細分化して再翻訳**

再翻訳時は、問題のチャンクをさらに小さく分割（100行以下）して再実行する。

### 6. ファイル結合

```bash
# 番号順に結合
cat temp_head.md temp_body_1.md temp_body_2.md ... temp_tail.md > "<filename>"
rm -f temp_head.md temp_body_*.md temp_tail.md
```

### 7. 徹底的レビュー（必須）

結合後、必ず以下の多角的レビューを実施する。並列Agent翻訳ではチャンク間で訳語・書式が不整合になるため、このステップで統一する。

#### 7.1 構造整合性チェック

原文と翻訳ファイルの構造要素数を比較する。

```bash
FILE="<翻訳ファイル>"
ORIG="<原文バックアップ>"
echo "総行数: $(wc -l < "$FILE") (原文: $(wc -l < "$ORIG"))"
echo "コードブロック: $(grep -c '^```' "$FILE") (原文: $(grep -c '^```' "$ORIG"))"
echo "H1: $(grep -c '^# ' "$FILE") (原文: $(grep -c '^# ' "$ORIG"))"
echo "H2: $(grep -c '^## ' "$FILE") (原文: $(grep -c '^## ' "$ORIG"))"
echo "H3: $(grep -c '^### ' "$FILE") (原文: $(grep -c '^### ' "$ORIG"))"
echo "H5: $(grep -c '^##### ' "$FILE") (原文: $(grep -c '^##### ' "$ORIG"))"
echo "画像: $(grep -c '!\[' "$FILE") (原文: $(grep -c '!\[' "$ORIG"))"
echo "章セパレータ: $(grep -c '^---$' "$FILE")"
echo "箇条書き: $(grep -c '^- ' "$FILE") (原文: $(grep -c '^- ' "$ORIG"))"
```

**期待値:** コードブロック数・見出し数・画像数は完全一致。総行数は80%以上、箇条書きは±数件以内。

#### 7.2 ラベル翻訳の統一

並列Agentで翻訳すると、`NOTE` / `TIP` / `WARNING` などのラベル翻訳がブレやすい。残存検出と統一を行う。

```bash
# 残存する英語ラベル
grep -nE "^(##### |\*\*)(NOTE|TIP|WARNING|Note|Tip|Warning)" "$FILE"
# 混在パターン（カッコ書きなど）
grep -nE "(注意（NOTE）|ヒント（TIP）|警告（WARNING）)" "$FILE"
```

検出された場合は、対象書のラベル形式（H5 か bold か）に揃えて統一する。

**ソースが装飾ラベル（Figure/Listing/Table）を H5 (`#####`) で出力している場合は注意**: 残すとObsidian等のアウトラインを破綻させる。bold (`**...**`) への変換を強く推奨。Manning livebook が代表例であり、専用スキル `manning-translator` に詳細な変換手順がある。

#### 7.3 主要用語の訳語統一

並列Agent間で同じ英語用語が異なる日本語に訳されることがある。主要用語の混在をチェック。

```bash
# observability の訳語ゆれ
echo "オブザーバビリティ: $(grep -c 'オブザーバビリティ' "$FILE")"
echo "観測可能性: $(grep -c '観測可能性' "$FILE")"
echo "可観測性: $(grep -c '可観測性' "$FILE")"
# telemetry の訳語ゆれ
echo "テレメトリ: $(grep -c 'テレメトリ' "$FILE")"
echo "テレメトリー: $(grep -c 'テレメトリー' "$FILE")"
# 他: latency/レイテンシ・レイテンシー、monitoring/モニタリング・監視 など
```

検出例（オブザーバビリティに統一）:

```bash
sed -i.bak \
  -e 's/観測可能性/オブザーバビリティ/g' \
  -e 's/可観測性/オブザーバビリティ/g' \
  -e 's/テレメトリー/テレメトリ/g' \
  "$FILE"
rm -f "${FILE}.bak"
```

**注意:** 動詞として使われる「監視する」のような自然な日本語表現は、カタカナ用語に変換しない。

#### 7.4 サービス名・固有名詞の統一

英語名と日本語訳が混在しやすい固有名詞をチェック。

```bash
echo "Model Service: $(grep -c 'Model Service' "$FILE")"
echo "Modelサービス: $(grep -c 'Modelサービス' "$FILE")"
echo "Session Service: $(grep -c 'Session Service' "$FILE")"
echo "セッションサービス: $(grep -c 'セッションサービス' "$FILE")"
# 他: Data Service, Tool Service, Observability Service, Workflow Service, Experimentation Service
```

ドミナントなパターン（多い方）に統一。通常は英語名が優勢なため、日本語訳を英語形に変換する:

```bash
sed -i.bak \
  -e 's/Modelサービス/Model Service/g' \
  -e 's/セッションサービス/Session Service/g' \
  "$FILE"
rm -f "${FILE}.bak"
```

**注意:** 「Observability Service（オブザーバビリティサービス）」のようなカッコ書き併記は意図的なものなので、これを壊さないよう確認する。

#### 7.5 ラベル行・装飾要素の統一

ソースに頻出する装飾ラベルやUI文字列が、Agentによって表記揺れすることがある（例: `livebook features:` → `livebook機能:` `livebookの機能:` など）。

```bash
# 装飾ラベル系の検出
grep -inE "^##### " "$FILE" | sed 's/[0-9].*$//' | sort -u
```

検出されたら統一する。原則として原文の表記（英語固有名詞は英語のまま）に揃える。

Manning livebook 等の特定ソース固有のラベル処理は `manning-translator` スキルを参照。

#### 7.6 章末セパレータと章ヘッダーの確認

各章末に `---` セパレータがあるか、章タイトルが「第N章」形式か確認。

```bash
grep -nE "^---$" "$FILE"        # セパレータ数 = 章数
grep -nE "^# 第[0-9]+章" "$FILE" # 章数
grep -nE "^# " "$FILE"          # H1全体（前書き含む）
grep -nE "^### 本章で扱う内容" "$FILE" # 各章の冒頭にあるか
```

#### 7.6.5 コードブロック破損の検出（翻訳後）

翻訳前検査（1.5）で見つけた抽出エラーが翻訳後も残っていることを確認。プレースホルダー化していない場合は警告:

```bash
# 数字のみ行を再カウント
COLLAPSED=$(awk '/^```/{c++; if(c%2==1){inblock=1; next} else {inblock=0; next}} inblock && /^[[:space:]]*[0-9]+[[:space:]]*$/' "$FILE" | wc -l)
echo "残存する破損コード行: $COLLAPSED"

# 該当箇所を一覧
awk '/^```/{c++; if(c%2==1){inblock=1; next} else {inblock=0; next}} inblock && /[0-9]{10,}/{print NR": "$0}' "$FILE"
```

検出されたら、ユーザーへ「N箇所のコードブロックが抽出エラーで失われている可能性があります。手動復元または再抽出を推奨」と報告する。

#### 7.7 コード内英語の確認

コードブロック内の英語（システムプロンプト、変数名等）は**翻訳しない**が、Agentが翻訳してしまうことがある。コード前後の文脈を読んで判断する。

```bash
# 翻訳されるべきでない英語が章本文に混入していないかチェック
awk '/^```/{c++; if(c%2==1){inblock=1; next} else {inblock=0; next}} inblock==0 && /^(The |This |That |These |Those |When |Where |If |For |To |As |However)/{print NR": "$0}' "$FILE"
```

該当行がコードブロック外で見つかった場合は翻訳漏れの可能性が高い。

#### 7.8 レビュー結果のユーザー報告

レビュー後、以下の情報を報告する:
- **構造整合性テーブル**: 翻訳・原文の各要素数の対比
- **修正した不整合カテゴリ**: 何を、何件、どう統一したか
- **確認済み・問題なしの項目**: コード内英語、章末セパレータなど

### 8. 最終検証と報告

```bash
grep -n "^# 第" "<filename>"
wc -l "<filename>"
```

ユーザーに報告:
- 翻訳した章の一覧
- **各章の原文行数と翻訳行数の対比テーブル**（比率も含む）
- 最終ファイルの総行数
- **レビューで修正した不整合のサマリー**

## 翻訳ルール概要

詳細は `references/translation-rules.md` を参照。

- 原文に忠実に翻訳。**内容の削除・要約・追加は絶対禁止**
- `#` 章タイトル（「第N章」形式）、`##` セクション、`###` サブセクション
- 技術用語: そのまま残すか初出時にカッコ書きで原語併記
- コード・JSON・SQL・擬似コード: そのまま保持
- テーブル・図キャプション・脚注: 翻訳する
- 各章の間に `---` セパレータ
