# 翻訳ルール詳細

## 最重要原則: 絶対に要約しない

**原文のすべての段落、すべての文、すべての箇条書き項目を翻訳すること。**

以下は明示的に禁止:
- 段落の省略や統合
- 「原文を参照してください」のような省略表現
- 箇条書き項目の削減
- コードブロックの省略（「長いため省略」等は禁止）
- テーブルの行の削減
- 脚注の省略

**検証基準: 翻訳後の行数が原文行数の80%未満なら、要約が発生している。再翻訳が必要。**

## 基本原則

- **原文忠実**: 原文の内容を削除・要約・追加しない。段落の順番、箇条書きの数、脚注もすべて原文通り
- **技術的正確性**: 技術概念の本質が伝わる自然な日本語にする。直訳で意味が通じにくい場合は、技術的に正確な意訳を行う
- **読みやすさ**: 冗長な敬語は避け、「〜です・〜ます」調で統一

## マークダウン構造

### 見出しレベル

| 原文パターン | 翻訳形式 | 例 |
|---|---|---|
| `Chapter N. Title` | `# 第N章 タイトル` | `# 第5章 構造化イベントはオブザーバビリティの構成要素` |
| セクション見出し | `## セクション名` | `## オブザーバビリティの数学的定義` |
| サブセクション | `### サブセクション名` | `### ソフトウェアの信頼性の特性` |

### 定型セクション

| 原文 | 翻訳 |
|---|---|
| `A NOTE FOR EARLY RELEASE READERS` | `早期リリース版読者の方へ` |
| `Conclusion` | `まとめ` または `結論` |
| `NOTE` / `WARNING` / `TIP` | `注記` / `警告` / `ヒント` |
| `Sidebar: Title` | `サイドバー：タイトル` |

### 装飾ラベルが H5 になっているソースへの一般原則

原文中で `Figure 1.1` `Listing 1.1` `NOTE` などの装飾ラベルが H5 (`##### ...`) として書かれている場合、そのまま訳すと Obsidian等のアウトラインを破綻させる。**H5 (`#####`) は章本文の小見出しのみに予約し、装飾ラベルは bold (`**...**`) に変換する**のが原則。

具体的な変換ルール（Listing/Figure/Table のフォーマットや UI 残骸など）はソースごとに異なるため、抽出元固有のスキルがあればそれに従う:

- Manning livebook / MEAP → `manning-translator` スキル
- それ以外 → 個別判断、または一般的な bold 変換

### 章間セパレータ

各章の間に `---`（水平線）を挿入する。

### コードブロック破損（抽出エラー）への一般対処

Web から markdown 抽出された原文では、コードブロック内のコード本体が失われ、`1234567891011121314` のような行番号連結数字のみが残ることがある。**数字列をそのまま翻訳結果に残してはいけない**（読者にとって無意味なノイズ）。

対処の選択肢:
1. 原文を再抽出
2. コードブロック全体を以下のプレースホルダーに置き換える:

```html
<!-- 注: 原文ではここに数行のコードが含まれていたが、抽出エラーのため省略 -->
```

3. 翻訳後に文脈から等価コードを再構築（過去事例: コミット「抽出エラーで失われたコード31箇所を文脈に基づき再構築」）

Manning livebook には固有の救済パターン（`manning-translator` スキルに自動化スクリプトあり）がある。

## コード・データの扱い

### コードブロック

- **コード自体はすべてそのまま保持（省略禁止）**
- コード内のコメントは日本語に翻訳してよい
- コード前後の説明文は翻訳する

```go
// 例: コメントは翻訳、コードはそのまま
func isBurnViolation(...) bool {
  // エラーバジェットが消耗されるかどうかを計算
  ...
}
```

### テーブル

マークダウンテーブル形式で再現する。ヘッダーと内容は翻訳する。**行を省略しない。**

### 図のキャプション

`Figure X-X.` は `図X-X.` に変換し、キャプション文は翻訳する。

### 脚注

- 脚注番号とリンクはそのまま保持
- 脚注のテキスト（書籍名、論文名は原語のまま、説明文は翻訳）
- 章末にまとめて配置

## Agent指示テンプレート

Agentに翻訳を委任する際、以下のテンプレートを使用する。**すべての項目を含めること。**

```
あなたは技術書の翻訳者です。以下のファイルのN行目〜M行目（約X行）を日本語に翻訳してください。

ファイルパス: <filepath>

【最重要ルール】
- 原文のすべての段落、すべての文、すべての箇条書き項目を翻訳すること
- 絶対に要約・省略しないこと。「原文を参照」のような省略は禁止
- コードブロックはすべてそのまま含めること（省略禁止）
- テーブルの行をすべて含めること
- 翻訳後の行数が原文行数（約X行）の80%以上になること。これを下回る場合は要約が発生している

【翻訳ルール】
- 原文に忠実に翻訳し、内容を削除・要約・追加しない
- マークダウン形式で構造化（# 章タイトル、## セクション、### サブセクション）
- 技術用語はそのまま残すか、初出時にカッコ書きで原語を併記
- コードブロック、JSON、テーブルはそのまま保持
- 「A NOTE FOR EARLY RELEASE READERS」→「早期リリース版読者の方へ」
- 章タイトルは「第N章」形式
- 各章の間に --- セパレータ

【コードブロック破損への対処 - 必須】
コードブロック内に `1234567891011121314` のような連続数字（行番号の連結）しか入っていない場合、それは抽出エラーでコード本体が消失している。以下に置き換える:

````
<!-- 注: 原文ではここに数行のコードが含まれていたが、抽出エラーのため省略 -->
````

連続数字をそのまま翻訳結果に残してはいけない（無意味なノイズになるため）。コード本体が正しく入っているコードブロックは原文通り保持する。

【出力指示】
翻訳結果を temp_body_<番号>.md に書き込んでください。
書き込み完了後、翻訳した行数を報告してください。
```

ソース固有の追加ルール（Manning livebook の装飾ラベル変換、livebook UI ブロック等）はそれぞれのスキルのテンプレートで補足する。

## 品質検証チェックリスト

翻訳完了後、以下を確認:

1. **行数比率**: 翻訳行数 ÷ 原文行数 ≧ 0.80
2. **章見出し数**: 原文の `Chapter` 数 = 翻訳の `# 第` 数
3. **セクション見出し数**: 原文のセクション見出し数 ≒ 翻訳のセクション見出し数
4. **コードブロック数**: 原文の ``` 数 = 翻訳の ``` 数
5. **箇条書き数**: 原文の `- ` 行数 ≒ 翻訳の `- ` 行数
6. **脚注数**: 原文の脚注数 = 翻訳の脚注数

検証に失敗した場合は、該当箇所を特定して再翻訳する。
