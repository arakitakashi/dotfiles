# ハンズオン教材作成スキル

特定技術について、最新の公式ドキュメントに基づいた実践型ハンズオン教材をマークダウンで作成する。学習者が自らコードを書き、技術の本質を深く理解することを目指す。

## 作成フロー

### Step 1: 要件の確認

以下を確認・決定する（ユーザーが未指定の場合は質問する）:

- **対象技術**: 何を学ぶか
- **レベル**: 初級 / 中級 / 上級（レベル別の設計方針は `references/level-guide.md` を参照）
- **ゴール**: ハンズオン完了時に何ができるようになるか
- **出力先ディレクトリ**: デフォルトはプロジェクトルート直下

### Step 2: 公式ドキュメントの徹底調査

**教材の信頼性は調査の質で決まる。** 執筆前に最新の公式ドキュメントを網羅的に調査する。

1. **context7 MCP サーバー**でライブラリ/フレームワークのドキュメントを取得
2. 不足があれば **WebSearch / WebFetch** で補完:
   - 公式ドキュメント、API リファレンス、ベストプラクティスガイド
   - リリースノート、最新の仕様変更
   - 公式チュートリアル（参考にするが丸写しはしない）
3. バージョンを明記する（例: 「React 19 時点」）

### Step 3: カリキュラム設計

調査結果に基づき、以下の構成でカリキュラムを設計する。**ユーザーの承認を得てから執筆に進む。**

```markdown
# [技術名] ハンズオン — [レベル]

## メタ情報
- 対象技術: [名前 + バージョン]
- レベル: [初級 / 中級 / 上級]
- 前提知識: [必要な知識]
- 所要時間目安: [時間]
- ゴール: [完了時にできること]

## カリキュラム
- Chapter 1: [タイトル] — [学習目標]
- Chapter 2: [タイトル] — [学習目標]
- ...
- Final Challenge: [タイトル] — [総合演習の概要]
```

**設計原則:**
- 概念の依存関係を分析し、前提知識なしで進められる順序にする
- 1チャプター1概念に絞る
- 各チャプターで「手を動かす」演習を必ず含める
- 最終チャプターは学んだ知識を統合する総合演習

### Step 4: セットアップファイルの生成（必要な場合）

学習者の不要な作業を減らすため、以下のようなセットアップファイルを生成する:

- プロジェクトの初期設定ファイル（`package.json`、`pyproject.toml` 等）
- 環境設定ファイル（`.env.example`、設定テンプレート等）
- テスト用のモックデータやフィクスチャ
- `setup.sh` / `setup.ps1`（環境構築の自動化）

**注意:** テーマの技術に関するコードは生成しない。学習者が自ら実装する。

### Step 5: ハンズオン教材の執筆

承認されたカリキュラムに基づき、チャプターごとに執筆する。

#### チャプターの構成

```markdown
## Chapter N: [タイトル]

### 学習目標
このチャプターで学ぶこと（1-2行）

### 背景知識
[概念の説明。なぜこの技術/パターンが必要かを実務の文脈で説明]

> 参考: [公式ドキュメントのタイトル](URL)

### 演習: [演習タイトル]

#### 要件
[学習者が実装すべき内容を明確に記述]

#### ステップ
1. [具体的な手順]
2. [具体的な手順]
...

#### 動作確認
[期待される出力や確認方法]

#### ヒント
<details>
<summary>ヒント 1: [ヒントのテーマ]</summary>
[ヒントの内容]
</details>

### 解説
[実装のポイント、なぜこのアプローチが良いのかの説明]

### チャレンジ課題（オプション）
[発展的な課題。余力がある学習者向け]

### このチャプターで学んだこと
- ポイント1
- ポイント2
- ポイント3
```

#### 執筆ルール

1. **公式ドキュメント準拠**: 説明は公式ドキュメントの内容に基づく。引用元を `> 参考: [タイトル](URL)` で明記
2. **自分で書かせる**: 核となるコードは学習者が実装する。完成コードの提示は最小限にし、ヒントは `<details>` で折りたたむ
3. **動作確認を必ず挟む**: 各演習後に「正しく動いているか」の確認ステップを入れる
4. **エラーへの備え**: よくあるエラーと対処法を NOTE として記載
5. **実務との接続**: 「実際のプロジェクトではこう使う」という文脈を添える
6. **段階的な難易度**: チャプターが進むにつれヒントを減らし、自律度を上げる

### Step 6: 総合演習の作成

最終チャプターとして、学んだ知識を統合する総合演習を作成する。

```markdown
## Final Challenge: [タイトル]

### シナリオ
[実務に近い具体的なシナリオを提示]

### 要件
[機能要件を箇条書きで列挙]

### 制約
[使用する技術、パターンなどの制約]

### 評価ポイント
- [ ] 要件1が満たされている
- [ ] 要件2が満たされている
- ...

### 解答例へのアプローチ
<details>
<summary>設計のヒント</summary>
[アーキテクチャや設計方針のヒント]
</details>

<details>
<summary>実装のヒント</summary>
[具体的な実装のヒント]
</details>
```

### Step 7: 付録の作成

以下のうち適切なものを含める:

- **クイックリファレンス**: よく使う API / コマンドの早見表
- **用語集**: ハンズオンで登場した用語の定義
- **トラブルシューティング**: よくあるエラーと対処法
- **次のステップ**: さらに学ぶべきテーマと推奨リソース
- **参考文献**: 公式ドキュメント URL の一覧

## 品質基準

- [ ] 執筆前に**公式ドキュメントの網羅的な調査**が行われている
- [ ] 引用元が **URL 付きで明記**されている（最低3箇所以上）
- [ ] 対象技術の**バージョンが明記**されている
- [ ] 各チャプターが**1概念に絞られている**
- [ ] 各チャプターに**手を動かす演習**が含まれている
- [ ] 演習の核となるコードは**学習者が自ら実装する**構成になっている
- [ ] ヒントが**折りたたみ**で提供されている
- [ ] **動作確認ステップ**が各演習に含まれている
- [ ] **総合演習**が最終チャプターに含まれている
- [ ] レベルに応じた**適切な難易度**になっている
- [ ] セットアップファイルで**不要な環境構築作業が削減**されている
